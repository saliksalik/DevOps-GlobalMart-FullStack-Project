Project Phoenix Story (Detailed + Easy Wording)

Salik got a clear business problem from leadership:
The GlobalMart e-commerce platform needed safer releases, fewer manual mistakes, and faster debugging when something failed.

At first, this looked like a coding task.
But after checking the situation, Salik understood the real issue was bigger than code.
The team had delivery problems:
- too many manual steps,
- risky deployments,
- weak visibility,
- and no smooth rollback path.

So Salik treated this as a full DevOps mission, not just an app run command.

Step 1: Understand the real pain
Salik started by identifying what was actually failing in day-to-day operations.
He found that most pain came from release process and operations, not just application logic.

Key learning:
If delivery process is weak, even good code becomes unstable in production.

Step 2: Stabilize the app first
Before touching DevOps tools, Salik verified the app itself.
He ran the API locally, ran tests, and checked the health endpoint.

Why this mattered:
If the base app is broken, CI/CD and infrastructure automation only fail faster.

Step 3: Containerize and run the local platform
Salik used Docker Compose to run the local stack.
This was not fully smooth in first try.
He had to fix container conflicts, startup issues, and build context problems.

Key learning:
DevOps work is not only writing YAML.
A big part is troubleshooting real runtime issues.

Step 4: Build a real CI pipeline with Jenkins
Then Salik set up Jenkins locally using Docker.
He faced multiple practical issues, one by one:
- Pipeline option missing,
- agent identity mismatch,
- plugin compatibility errors,
- Jenkinsfile parsing/runtime errors,
- shell substitution issue in pipeline scripts.

He fixed each issue step by step:
- installed required plugins,
- simplified Jenkinsfile for local environment,
- removed unsupported options,
- corrected shell variable usage,
- reran builds until pipeline became stable.

Result:
Pipeline completed successfully.

Big learning:
CI/CD is iterative. You run, debug, patch, and rerun until reliable.

Step 5: Build observability (monitoring + logging)
Salik validated the full observability path:
- API health endpoint,
- Prometheus reachable,
- Grafana reachable,
- Kibana reachable,
- Elasticsearch index and count present.

He generated both success and failure traffic:
- 200 response paths,
- expected 404 path.

He also sent synthetic JSON logs to Logstash and verified they were indexed in Elasticsearch.

Why this was important:
Strong teams monitor both happy path and failure path.

Step 6: Why 404 testing was important
Salik intentionally called a non-existing product ID.
The API returned 404 correctly.

This proved:
- error handling works,
- logs and metrics capture real-world bad requests,
- system behavior is predictable under failure scenarios.

Step 7: Move to Kubernetes and blue-green deployment
After local platform and observability were stable, Salik enabled local Kubernetes (Docker Desktop cluster).
He deployed manifests and then handled image pull failures by switching Kubernetes manifests to local images.

Then he executed blue-green deployment flow:
- detected live and idle slots,
- deployed to idle slot,
- waited for health,
- switched service traffic to new slot,
- verified live slot changed to green,
- scaled down blue deployment to complete cleanup.

Result:
Blue-green flow worked end-to-end on local cluster.

Current state summary
- App validated
- Docker stack validated
- Jenkins CI pipeline validated
- Monitoring and logging validated
- Kubernetes deployment validated
- Blue-green traffic switch validated
- Old slot cleanup completed

One-line story summary
Salik turned a "just run the app" assignment into a full DevOps transformation journey: stable app baseline -> reliable CI pipeline -> complete observability -> Kubernetes deployment -> safe blue-green release and cleanup.
